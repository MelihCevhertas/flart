# v0.3.0 manual test plan

End-to-end scenarios the author (or any maintainer) should walk through in
a real Claude Code session *before* publishing the draft GitHub release.
The unit + integration suite (`tools/test_all.sh`) covers the filter logic,
hook installer, and CLI plumbing; this plan exercises the live wiring
between Claude Code v2.1.121+ and the installed hooks.

**Test host requirements**
- Claude Code v2.1.121 or newer (verify with `claude --version`).
- A real Flutter project on disk — the Wonderous repo is the reference
  bed, but any project with `pubspec.yaml` works.
- `jq` on `$PATH`.
- A fresh shell where `flart init` has just been re-run with the v0.3.0
  binary (so all three hook scripts are present).

Each scenario has: a **prep** step (what to run from the agent's prompt
or your shell), a **verify** step (what to check), and an **expected**
note (what success looks like).

---

## 1. `flart analyze` — PreToolUse / Bash rewrite (unchanged from v0.1.0)

Sanity check the existing flutter→flart rewrite path still works.

- **Prep:** In a Claude Code session, ask the agent to "run flutter
  analyze". The PreToolUse / Bash hook should rewrite the command.
- **Verify:** `flart savings --by-command` shows a new `analyze` row;
  the agent's transcript shows the compact analyzer summary (warnings
  grouped by rule), not the raw `dart analyze --format=machine` output.
- **Expected:** ~95%+ byte compression. The agent never sees the
  permission prompt for the rewritten command.

## 2. `grep -rn "TODO" lib/` — PostToolUse / Bash filter (new in v0.3.0)

The headline v0.3.0 feature: raw Bash output gets mediated.

- **Prep:** Ask the agent to "find all TODO comments in lib/". Watch it
  run `grep -rn "TODO" lib/` (or similar).
- **Verify:**
  - The agent's transcript shows a filtered view that opens with
    `Output filtered by flart: showing first 20 + last 5 of N lines.`
    (or `first 15 + last 5 of N` for outputs > 200 lines) followed by
    the tail and an `Error-flavoured lines from the elided body` block.
  - The footer reads `Full log: ~/.local/share/flart/tee/<epoch>_grep_bash.log`.
  - `flart savings --by-command` now lists a `grep` row under module
    `bash_post`.
- **Expected:** Byte savings reported in `flart savings`; the agent
  reasons over the head + tail + errors without paying for the body.

## 3. Tee recovery via `cat` — heuristic bypass

Verify the recovery path doesn't loop back through the filter.

- **Prep:** Grab a tee path from scenario 2's footer. Ask the agent
  (or just run yourself in the same session) `cat <tee path>`.
- **Verify:** The full file content appears in the agent's transcript,
  not a filtered view. `flart savings` shows **no new** `bash_post`
  row (the bypass branch doesn't record).
- **Expected:** The bypass-reason path covers `cat | head | tail | less |
  more | wc | grep` against any arg under
  `~/.local/share/flart/tee/`. Confirms the heuristic catch.

## 4. `FLART_FULL_OUTPUT=1` — explicit escape

Belt-and-braces alternative to the tee-read heuristic.

- **Prep:** Run `FLART_FULL_OUTPUT=1 find lib -name "*.dart"` in the
  agent session (or your shell — the hook fires regardless of who
  invoked it).
- **Verify:** Full `find` output reaches the agent's transcript verbatim;
  no `Output filtered by flart` header. No new `bash_post` row in savings.
- **Expected:** The env-prefix escape works without the user needing to
  remember tee paths. Documented in CLAUDE.md so the agent can suggest
  it when the user asks for full output.

## 5. Failed command — error preservation

The plan-v1.15 format: framed stderr + stdout tail + tee path.

- **Prep:** Run `ls /nonexistent_path` in the agent session.
- **Verify:** The agent sees either:
  - the framed `Command failed (exit 2). stderr (… KB): …` template
    (when output is substantial enough that framing compresses), **or**
  - the raw Claude Code default formatting (when the anti-bloat
    fallback kicks in for tiny errors, e.g. a single-line stderr).
- **Expected:** Error context (stderr) is never silently dropped; the
  stdout tail is preserved for trace context; the tee path is referenced
  when a full log was captured. Either branch is acceptable — both
  preserve the failure semantics, just at different verbosity.

## 6. Claude Code version detection — old version warning

Force a downgrade via a fake `claude` shim and verify the gate.

- **Prep:**
  ```bash
  mkdir -p /tmp/fake-claude
  cat > /tmp/fake-claude/claude <<'EOF'
  #!/usr/bin/env bash
  echo "Claude Code v2.1.119 (fake)"
  EOF
  chmod +x /tmp/fake-claude/claude
  PATH="/tmp/fake-claude:$PATH" flart init --check
  ```
- **Verify:** The check table includes a `✓ Claude Code` row whose
  detail reads `2.1.119 (output mutation requires 2.1.121+)` and a hint
  `→ Upgrade Claude Code to 2.1.121+ to enable PostToolUse / Bash output
  mutation`. The `Hook script (Bash, PostToolUse)` row is `✓ … skipped
  (Claude Code < 2.1.121)` — not a failure.
- **Expected:** The installer (re-)gate doesn't break older Claude
  installs; users get an actionable upgrade nudge rather than a silent
  half-install. Cleanup: `rm -rf /tmp/fake-claude`.

## 7. Reporter — `bash_post` shows up alongside `filter` and `executor`

After running 1, 2, and 5 above:

- **Prep:** `flart savings --by-module`
- **Verify:** The table includes three modules: `filter` (from analyze),
  `executor` (if you also ran `flart exec` during the session), and
  `bash_post` (from the grep + make scenarios). Each row reports
  invocation count + token savings.
- **Expected:** No code change was needed to surface `bash_post` —
  the aggregator's `byModule` query groups dynamically. If the row is
  missing, scenarios 2 and 5 didn't trigger mutation; check the bypass
  reasons in `flart savings --details`.

## Smoke binary build (sanity, before publishing)

```bash
dart compile exe packages/flart_cli/bin/flart.dart -o /tmp/flart-smoke \
  --define=FLART_VERSION=0.3.0-rc \
  --define=GIT_SHA=$(git rev-parse --short HEAD) \
  --define=BUILD_DATE=$(date -u +%Y-%m-%d)

/tmp/flart-smoke version           # → flart 0.3.0-rc (commit …, built …)
/tmp/flart-smoke init --check      # → Claude Code row + all hook scripts
/tmp/flart-smoke savings --by-module   # → shows bash_post if you've run
                                       #   a real session against this binary

# Synthetic stdin smoke for the PostToolUse pipeline:
echo '{"tool_input":{"command":"grep TODO lib/"},"tool_response":{"stdout":"'"$(seq 1 100)"'","stderr":"","exit_code":0},"cwd":"/tmp"}' \
  | /tmp/flart-smoke bash-post-hook | jq .
# Expect: hookSpecificOutput.updatedToolOutput containing
# "Output filtered by flart: showing first 20 + last 5 of 100 lines."

rm -f /tmp/flart-smoke
```

If any scenario above misbehaves, **do not publish the draft release.**
Roll back the tag (`git tag -d v0.3.0 && git push origin :refs/tags/v0.3.0`)
and iterate. The release flow is forgiving — drafts are private until you
hit "Publish".
